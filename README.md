# Sticky Board

Sticky Board is a full-stack productivity web app for private sticky notes and Kanban-style task management. It uses React, TypeScript, TanStack Start, Vite, Tailwind CSS, and Lovable Cloud for authentication, database, and deployed backend services.

## Current app features

- Email/password authentication and Google sign-in
- Per-user private sticky notes
- Create, edit, pin, delete, search, sort, and color-code notes
- Drag-and-drop note reordering
- Kanban task board with three stages: **To Do**, **Ongoing**, **Completed**
- Drag-and-drop tasks between columns
- Task priorities: **Low**, **Medium**, **High**, **Urgent**
- Priority guide explaining what each level means
- Due dates, overdue labels, and task counters
- Light/dark theme toggle
- Responsive layout for mobile, tablet, and laptop/desktop screens
- Backend tables protected by row-level security so users only see their own data

## Important architecture note

This repository is **not a plain static Vite SPA**. It is a TanStack Start app powered by Vite.

That means:

- There is intentionally **no `index.html`** file.
- The HTML shell is generated by `src/routes/__root.tsx`.
- Routes are file-based under `src/routes/`.
- `src/routeTree.gen.ts` is generated by the TanStack Router plugin and should not be edited manually.
- Do not run `npx vite`, `vite --host`, or another generic Vite command outside the npm scripts.
- Always use `npm run dev` locally.

If a cloned project shows `404 Page not found` for `/`, `/auth`, or `/tasks`, the most common cause is starting it as a plain Vite app instead of using the configured TanStack Start/Vite setup.

## Local setup, step by step

### 1. Install Node.js

Use Node.js 20 or newer.

```bash
node -v
```

If the version is below 20, install Node 20+ and reopen the terminal.

### 2. Clone and enter the project

```bash
git clone <your-repository-url>
cd <your-project-folder>
```

### 3. Install dependencies

```bash
npm install
```

### 4. Create `.env`

Create a `.env` file in the project root. Use the same public backend values from your Lovable project environment:

```env
VITE_SUPABASE_URL=your_backend_url
VITE_SUPABASE_PUBLISHABLE_KEY=your_publishable_key
VITE_SUPABASE_PROJECT_ID=your_project_id

SUPABASE_URL=your_backend_url
SUPABASE_PUBLISHABLE_KEY=your_publishable_key
SUPABASE_PROJECT_ID=your_project_id
```

Notes:

- The `VITE_` values are required in the browser.
- The non-`VITE_` values are used by the server/runtime side.
- Do not place private service-role secrets in client-visible `VITE_` variables.
- In Lovable, these values are already managed for preview and publishing.

### 5. Start local development

```bash
npm run dev
```

Then open the local URL printed in the terminal, usually:

```text
http://localhost:5173
```

### 6. Test the main routes

Open these directly in the browser:

```text
http://localhost:5173/
http://localhost:5173/auth
http://localhost:5173/tasks
```

All three should load. If you are not signed in, `/` and `/tasks` redirect to `/auth` from the app UI.

### 7. Build for production

```bash
npm run build
```

### 8. Preview the production build locally

```bash
npm run preview
```

or:

```bash
npm start
```

## Database setup

The backend schema is in `supabase/migrations/`.

Tables added by this project:

- `notes` вЂ” stores user-owned sticky notes
- `tasks` вЂ” stores user-owned Kanban tasks

Custom task types:

- `task_status`: `todo`, `ongoing`, `completed`
- `task_priority`: `low`, `medium`, `high`, `urgent`

Security model:

- Row-level security is enabled.
- Authenticated users can only read/write their own notes and tasks.
- Public anonymous users cannot access note/task data.

## Special extras added for review

Use this section when presenting the project:

1. **Full authentication flow**
   - Users can sign up, sign in, sign out, and use Google sign-in.
   - Data is scoped to the authenticated user.

2. **Sticky notes workspace**
   - Notes support title, content, color, pinning, search, sorting, deletion with undo, and drag ordering.

3. **Task management board**
   - Tasks are organized into To Do, Ongoing, and Completed columns.
   - Drag-and-drop updates task status.

4. **Priority system**
   - Low, Medium, High, and Urgent priorities are visually labeled.
   - The priority guide explains how to choose each priority.

5. **Responsive UI**
   - Layout adapts for phones, tablets, laptops, and desktops.

6. **Theme support**
   - The app includes light and dark mode.

7. **Secure backend model**
   - Notes and tasks use authenticated access and database-level row security.

8. **Production-oriented routing**
   - Routes are declared with TanStack file-based routing and include metadata for SEO/social previews.

9. **SSR failure protection**
   - `src/server.ts`, `src/lib/error-capture.ts`, and `src/lib/error-page.ts` provide safer error handling if the server render fails.

## Troubleshooting local 404 errors

Try these in order:

1. Stop the dev server.
2. Delete generated/build folders:

   ```bash
   rm -rf node_modules .tanstack .output dist
   ```

3. Reinstall:

   ```bash
   npm install
   ```

4. Start only with:

   ```bash
   npm run dev
   ```

5. Confirm these files exist:

   ```text
   src/routes/__root.tsx
   src/routes/index.tsx
   src/routes/auth.tsx
   src/routes/tasks.tsx
   vite.config.ts
   ```

6. Confirm `vite.config.ts` imports `defineConfig` from `@lovable.dev/vite-tanstack-config`.

7. Do not create `index.html`; that is not needed for this app type.

## Safe files/folders to delete locally

These can be deleted and regenerated:

- `node_modules/`
- `.tanstack/`
- `.output/`
- `dist/`
- `.nitro/`
- `.vinxi/`

Do **not** delete:

- `src/routes/`
- `src/router.tsx`
- `src/start.ts`
- `src/server.ts`
- `vite.config.ts`
- `supabase/migrations/`
- `src/integrations/supabase/`

## Deployment notes

- Lovable preview and publishing are already configured for this project.
- Backend/database changes are managed through Lovable Cloud migrations.
- Frontend changes require publishing/updating the app from Lovable when you want them live.
- For self-hosting, keep the TanStack Start setup intact and deploy as a server-rendered app, not as a static `index.html` SPA.
